## General Description

The final project is the synthesis of everything learned throughout the course. Each team will design and implement a complete parallel computing application: from the sequential reference version to an optimized parallel implementation with quantitative performance analysis.

The goal is not only for the program to work, but for students to be able to explain every design decision: why they chose that technology, how they distribute the work, what bottlenecks they found, and how they resolved them.

---

## Objectives

Upon completing the project, each student must be able to:

- Analyze a computational problem and identify the parallelizable parts.
- Design a parallelization strategy using at least two technologies from the course.
- Implement and debug parallel code in a real environment.
- Measure speedup and efficiency using a reproducible methodology.
- Quantitatively compare a sequential implementation against the parallel one.
- Communicate technical results clearly in a report and oral presentation.

---

## Project Requirements

### Mandatory technical requirements

1. **At least two parallel technologies** from the following set: MPI, OpenMP, Pthreads, CUDA.
2. **Sequential reference implementation** to calculate real speedup.
3. **Performance analysis** with at least three different problem sizes.
4. **Reported metrics**: execution time, speedup S(p) = T(1)/T(p), efficiency E(p) = S(p)/p.
5. **Scalability**: show how performance varies when changing the number of processes/threads/blocks.
6. **Error handling**: the code must not fail silently on invalid inputs or allocation errors.

### Team requirements

- Groups of **2 to 3 students**.
- Each member must be able to defend any part of the project during the presentation.
- The GitHub repository must show contributions from all members (commit history).

<Alert type="warning">
Projects with commits from only one member will be evaluated with a reduction factor applied to the team grade. Genuine collaborative work is expected — not a division where "one person implements and the other documents."
</Alert>

## Deliverables

### 1. Source code on GitHub

- Public repository or shared with the instructor.
- Clear folder structure: `src/`, `sequential/`, `parallel/`, `benchmarks/`, `report/`.
- `Makefile` or script that compiles the entire project with a single command.
- `README.md` with step-by-step compilation and execution instructions.
- Appropriate `.gitignore` (no compiled binaries or temporary files).

### 2. Technical report

PDF document of 8 to 15 pages. See the **Report Format** section below.

### 3. Oral presentation (15 minutes)

- 10 minutes of presentation + 5 minutes of questions.
- Must include: motivation, design, live demo, results, conclusions.
- All members must speak during the presentation.
- Free format: slides, Jupyter Notebook, terminal, whatever you prefer.

### 4. Live demo

During the presentation, you will run the program in real time. You must be able to:
- Change the problem size and show how the time varies.
- Show the GPU monitor (`nvidia-smi`) or CPU metrics during execution.
- Answer questions about the code directly in the editor or terminal.

<Alert type="warning">
The live demo is mandatory. A project that cannot be executed on the day of the presentation will receive a severe penalty in the rubric, regardless of the quality of the report.
</Alert>

## Report Format

The report must be written in English, in PDF format, with the following sections in order:

### Cover page
- Project title, full names of all members, course code, submission date.

### Abstract
Maximum 200 words. Briefly describe the problem, the parallelization approach, the main results, and the conclusions. It must be readable independently of the rest of the report.

### 1. Introduction
- Problem description: what is being solved and why it is computationally expensive.
- Motivation for parallelization.
- Specific objectives of the project.

### 2. Methodology
- Description of the sequential algorithm.
- Parallelization strategy: how work is divided, what data is shared, how processes are synchronized.
- Technologies used and justification for why they were chosen.
- Test environment: hardware (CPU, GPU, RAM), operating system, compiler and library versions.

### 3. Results and Analysis
- Table of execution times for different problem sizes and number of processes/threads.
- Speedup and efficiency charts.
- Analysis of results: does the speedup approach the ideal? Where are the bottlenecks?
- Application of Amdahl's Law where applicable.

### 4. Conclusions
- What worked well and what was difficult.
- Limitations of the current implementation.
- Future work: what you would improve with more time.

### 5. References
Bibliography in APA format. Minimum 5 references (books, articles, or official documentation).

## Frequently Asked Questions

**Can I use external libraries such as Thrust or OpenBLAS?**
Yes, but you must understand what the library does. If you use `thrust::sort`, you must be able to explain what parallel algorithm it executes internally.

**Can the report be written in Spanish?**
No. The report and the presentation must be in English. The code may use variable names and comments in either language, which is standard convention.

**What if the team has only 2 people and the topic is High difficulty?**
The same level of quality is expected. The difficulty of the topic does not adjust the rubric, but it may result in a more impressive project if executed well.

**Does the repository have to be public?**
It can be private, but it must be shared with the instructor's GitHub user before the submission deadline.

<Alert type="tip">
Use GitHub Projects or Issues to distribute tasks within the team. This also serves as evidence of collaborative work and can compensate for any imbalance in code commits.
</Alert>
