## Introduction

Designing efficient parallel programs requires choosing the right parallelization strategy for each problem. There is no universal solution: the choice depends on the structure of the problem, data dependencies, and the architecture of the available hardware.

In this laboratory we will explore the main parallelization strategies, learn how to balance workloads across processes, and analyze the fundamental laws governing the performance of parallel systems.

Make sure you have MPICH or OpenMPI and GCC with Pthreads support installed. Verify with `mpicc --version` and `gcc --version`.
</Alert>

## 1. Data Decomposition

**Data decomposition** divides the dataset among the available processes so that each process works on its portion independently.

### Block Distribution

Each process receives a contiguous block of elements. Given N elements and P processes, process `p` receives elements from `p*(N/P)` to `(p+1)*(N/P) - 1`.

### Cyclic Distribution

Elements are assigned in a round-robin fashion. Process `p` receives elements `p, p+P, p+2P, ...`. This distribution improves load balancing when the processing cost varies per element.

### Block-Cyclic Distribution

Combines both: blocks of size `b` are distributed cyclically. This is the strategy used by ScaLAPACK for dense matrices.

### Distribution Table (N=12, P=4)

| Process | Block | Cyclic | Block-Cyclic (b=2) |
|---------|-------|--------|--------------------|
| 0 | 0, 1, 2 | 0, 4, 8 | 0, 1, 8, 9 |
| 1 | 3, 4, 5 | 1, 5, 9 | 2, 3, 10, 11 |
| 2 | 6, 7, 8 | 2, 6, 10 | 4, 5 |
| 3 | 9, 10, 11 | 3, 7, 11 | 6, 7 |

### Exercise 1: Vector Sum with Block vs. Cyclic Distribution

Implement the sum of a vector of N elements using both distributions and compare the execution time.

```c
#include <mpi.h>
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

#define N 1000000

double suma_bloque(int rank, int size, double *v) {
    int inicio = rank * (N / size);
    int fin    = (rank == size - 1) ? N : inicio + (N / size);
    double local = 0.0;
    for (int i = inicio; i < fin; i++)
        local += v[i];
    return local;
}

double suma_ciclica(int rank, int size, double *v) {
    double local = 0.0;
    for (int i = rank; i < N; i += size)
        local += v[i];
    return local;
}

int main(int argc, char **argv) {
    MPI_Init(&argc, &argv);

    int rank, size;
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &size);

    double *v = NULL;
    if (rank == 0) {
        v = malloc(N * sizeof(double));
        srand(42);
        for (int i = 0; i < N; i++)
            v[i] = (double)rand() / RAND_MAX;
    } else {
        v = malloc(N * sizeof(double));
    }

    MPI_Bcast(v, N, MPI_DOUBLE, 0, MPI_COMM_WORLD);

    double t0, local_sum, total_sum;

    t0 = MPI_Wtime();
    local_sum = suma_bloque(rank, size, v);
    MPI_Reduce(&local_sum, &total_sum, 1, MPI_DOUBLE, MPI_SUM, 0, MPI_COMM_WORLD);
    if (rank == 0)
        printf("Bloque   -> suma=%.4f  tiempo=%.6f s\n", total_sum, MPI_Wtime() - t0);

    t0 = MPI_Wtime();
    local_sum = suma_ciclica(rank, size, v);
    MPI_Reduce(&local_sum, &total_sum, 1, MPI_DOUBLE, MPI_SUM, 0, MPI_COMM_WORLD);
    if (rank == 0)
        printf("Cíclica  -> suma=%.4f  tiempo=%.6f s\n", total_sum, MPI_Wtime() - t0);

    free(v);
    MPI_Finalize();
    return 0;
}
```

Compile and run:

```bash
mpicc -O2 -o suma_distribuciones ejercicio1.c
mpirun -np 4 ./suma_distribuciones
```

<Alert type="tip">
Block distribution is usually more cache-efficient because it accesses memory contiguously. Cyclic distribution may be necessary when the processing cost per element is irregular.
</Alert>

## 2. Task Decomposition

**Task decomposition** identifies independent units of work (tasks) that can execute in parallel. It is modeled as a **Directed Acyclic Graph (DAG)** where nodes are tasks and edges represent dependencies.

### DAG Concept

```
Read (T1)
    |
    v
Process (T2) --> Statistics (T4)
    |
    v
Write (T3)
```

The **critical path** of the DAG determines the lower bound on parallel execution time.

### Exercise 2: Pipeline Pattern with Pthreads

Implement a three-stage pipeline: read, process, and write. Each stage is a thread that consumes the output of the previous stage.

```c
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

#define BUFFER_SIZE 8
#define TOTAL_ITEMS 32

typedef struct {
    int datos[BUFFER_SIZE];
    int cabeza, cola, count;
    pthread_mutex_t mutex;
    pthread_cond_t  no_lleno;
    pthread_cond_t  no_vacio;
} Buffer;

Buffer buf_leer_procesar;
Buffer buf_procesar_escribir;
int items_leidos    = 0;
int items_procesados = 0;

void buffer_init(Buffer *b) {
    b->cabeza = b->cola = b->count = 0;
    pthread_mutex_init(&b->mutex, NULL);
    pthread_cond_init(&b->no_lleno, NULL);
    pthread_cond_init(&b->no_vacio, NULL);
}

void buffer_poner(Buffer *b, int val) {
    pthread_mutex_lock(&b->mutex);
    while (b->count == BUFFER_SIZE)
        pthread_cond_wait(&b->no_lleno, &b->mutex);
    b->datos[b->cola] = val;
    b->cola = (b->cola + 1) % BUFFER_SIZE;
    b->count++;
    pthread_cond_signal(&b->no_vacio);
    pthread_mutex_unlock(&b->mutex);
}

int buffer_tomar(Buffer *b) {
    pthread_mutex_lock(&b->mutex);
    while (b->count == 0)
        pthread_cond_wait(&b->no_vacio, &b->mutex);
    int val = b->datos[b->cabeza];
    b->cabeza = (b->cabeza + 1) % BUFFER_SIZE;
    b->count--;
    pthread_cond_signal(&b->no_lleno);
    pthread_mutex_unlock(&b->mutex);
    return val;
}

void *hilo_leer(void *arg) {
    for (int i = 0; i < TOTAL_ITEMS; i++) {
        usleep(1000);
        buffer_poner(&buf_leer_procesar, i);
        printf("[Lectura]     item %d\n", i);
    }
    buffer_poner(&buf_leer_procesar, -1);
    return NULL;
}

void *hilo_procesar(void *arg) {
    while (1) {
        int val = buffer_tomar(&buf_leer_procesar);
        if (val == -1) { buffer_poner(&buf_procesar_escribir, -1); break; }
        int resultado = val * val;
        usleep(2000);
        buffer_poner(&buf_procesar_escribir, resultado);
        printf("[Procesamiento] item %d -> %d\n", val, resultado);
    }
    return NULL;
}

void *hilo_escribir(void *arg) {
    while (1) {
        int val = buffer_tomar(&buf_procesar_escribir);
        if (val == -1) break;
        printf("[Escritura]   resultado %d\n", val);
        usleep(500);
    }
    return NULL;
}

int main() {
    buffer_init(&buf_leer_procesar);
    buffer_init(&buf_procesar_escribir);

    pthread_t t1, t2, t3;
    pthread_create(&t1, NULL, hilo_leer,     NULL);
    pthread_create(&t2, NULL, hilo_procesar, NULL);
    pthread_create(&t3, NULL, hilo_escribir, NULL);

    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    pthread_join(t3, NULL);
    return 0;
}
```

## 3. Load Balancing

**Load balancing** aims to distribute work so that all processes finish at approximately the same time. An idle process while others are working wastes resources.

### Static Balancing

The load is assigned before running the program. It is simple but does not adapt to variations in processing time per element.

### Dynamic Balancing

Work is assigned at runtime. The **master-worker** pattern is the most common implementation: the master distributes units of work as workers become available.

### Exercise 3: Master-Worker Pattern with MPI

```c
#include <mpi.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

#define TOTAL_TAREAS 64
#define TAG_TAREA    1
#define TAG_RESULTADO 2
#define TAG_FIN      0

int procesar_tarea(int id) {
    usleep((id % 5 + 1) * 10000);
    return id * id;
}

void maestro(int size) {
    int proxima = 0;
    int activos = size - 1;
    int tarea, resultado, trabajador;
    MPI_Status estado;

    for (int w = 1; w < size && proxima < TOTAL_TAREAS; w++, proxima++) {
        MPI_Send(&proxima, 1, MPI_INT, w, TAG_TAREA, MPI_COMM_WORLD);
        printf("[Maestro] tarea %d -> trabajador %d\n", proxima, w);
    }

    while (activos > 0) {
        MPI_Recv(&resultado, 1, MPI_INT, MPI_ANY_SOURCE, TAG_RESULTADO,
                 MPI_COMM_WORLD, &estado);
        trabajador = estado.MPI_SOURCE;
        printf("[Maestro] resultado %d <- trabajador %d\n", resultado, trabajador);

        if (proxima < TOTAL_TAREAS) {
            MPI_Send(&proxima, 1, MPI_INT, trabajador, TAG_TAREA, MPI_COMM_WORLD);
            printf("[Maestro] tarea %d -> trabajador %d\n", proxima, trabajador);
            proxima++;
        } else {
            tarea = -1;
            MPI_Send(&tarea, 1, MPI_INT, trabajador, TAG_FIN, MPI_COMM_WORLD);
            activos--;
        }
    }
}

void trabajador(int rank) {
    int tarea, resultado;
    MPI_Status estado;

    while (1) {
        MPI_Recv(&tarea, 1, MPI_INT, 0, MPI_ANY_TAG, MPI_COMM_WORLD, &estado);
        if (estado.MPI_TAG == TAG_FIN) break;
        resultado = procesar_tarea(tarea);
        MPI_Send(&resultado, 1, MPI_INT, 0, TAG_RESULTADO, MPI_COMM_WORLD);
    }
}

int main(int argc, char **argv) {
    MPI_Init(&argc, &argv);
    int rank, size;
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &size);

    if (size < 2) {
        if (rank == 0) fprintf(stderr, "Se necesitan al menos 2 procesos.\n");
        MPI_Finalize();
        return 1;
    }

    if (rank == 0) maestro(size);
    else           trabajador(rank);

    MPI_Finalize();
    return 0;
}
```

```bash
mpicc -O2 -o maestro_trabajador ejercicio3.c
mpirun -np 5 ./maestro_trabajador
```

<Alert type="warning">
The master-worker pattern can become a bottleneck when the number of workers is very large. In that case, consider a hierarchy of masters.
</Alert>

## 4. Amdahl's Law vs. Gustafson's Law

### Amdahl's Law

Limits speedup when the problem size is **fixed**:

$$S(P) = \frac{1}{f_s + \frac{1 - f_s}{P}}$$

Where $f_s$ is the sequential fraction of the program and $P$ is the number of processors. The theoretical maximum speedup is $\frac{1}{f_s}$.

### Gustafson's Law

Models speedup when the problem size **grows** with $P$ (weak scaling):

$$S(P) = P - f_s \cdot (P - 1)$$

This law is more optimistic and better reflects real practice, where having more processors typically means solving larger problems.

### Comparison

| Aspect | Amdahl's Law | Gustafson's Law |
|--------|-------------|-----------------|
| Problem size | Fixed | Grows with P |
| Scaling type | Strong scaling | Weak scaling |
| Speedup outlook | Pessimistic | Optimistic |
| Use case | Reduce time for a fixed problem | Solve larger problems |
| Limit as $P\to\infty$ | $\frac{1}{f_s}$ | Unbounded |

### Exercise 4: Theoretical Speedup Calculation

Given a program with sequential fraction $f_s = 0.05$ (5%), compute the speedup according to both laws for $P = \{1, 2, 4, 8, 16, 32, 64\}$.

```c
#include <stdio.h>

int main() {
    double fs = 0.05;
    int procesadores[] = {1, 2, 4, 8, 16, 32, 64};
    int n = 7;

    printf("%-6s  %-12s  %-12s\n", "P", "Amdahl", "Gustafson");
    printf("%-6s  %-12s  %-12s\n", "---", "----------", "-----------");

    for (int i = 0; i < n; i++) {
        int P = procesadores[i];
        double amdahl    = 1.0 / (fs + (1.0 - fs) / P);
        double gustafson = P - fs * (P - 1);
        printf("%-6d  %-12.4f  %-12.4f\n", P, amdahl, gustafson);
    }
    return 0;
}
```

```bash
gcc -O2 -o speedup ejercicio4.c
./speedup
```

Observe how the gap between both laws widens as P grows.

## 5. Parallel Design Patterns

Parallel design patterns are reusable solutions to recurring problems in parallel programming.

| Pattern | Description | Typical use case |
|---------|-------------|-----------------|
| **Map** | Applies the same function to each element independently | Vector/image transformation |
| **Reduce** | Combines elements via an associative operation | Sum, maximum, global product |
| **Scan** | Computes parallel prefix sums | Compression, ranking, histograms |
| **Stencil** | Updates cells based on neighbors | PDE simulations, image filters |
| **Pipeline** | Chained stages, each processing the output of the previous | Stream processing, compilers |
| **Fork-Join** | Divides work into subproblems, solves in parallel, combines | Merge sort, parallel quicksort |

<Alert type="tip">
The **Map** and **Reduce** patterns are the foundation of frameworks like MapReduce and Spark. Understanding them gives you the intuition to scale any problem to clusters.
</Alert>
