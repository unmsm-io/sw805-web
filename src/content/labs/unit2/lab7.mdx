## Introducción

El modelo de paralelismo de bucles con `#pragma omp parallel for` funciona bien cuando el trabajo está distribuido en iteraciones regulares. Pero muchos algoritmos importantes —recursión, grafos, procesamiento de árboles— no encajan en ese molde. Para estos casos, OpenMP 3.0 introdujo el **modelo de tareas**.

En el modelo de tareas, el programador define unidades de trabajo (tareas) que el runtime de OpenMP encola y despacha dinámicamente entre los hilos disponibles. Esto permite paralelizar algoritmos irregulares sin necesidad de conocer de antemano cuánto trabajo hay ni cómo distribuirlo.

En este laboratorio exploraremos las directivas de tareas, la planificación dinámica de bucles y las secciones paralelas.

## Directivas de tareas

### `#pragma omp task`

Crea una tarea que será ejecutada por algún hilo del equipo actual. La tarea puede ejecutarse inmediatamente o encolarse para ejecución futura.

```c
#pragma omp parallel
{
    #pragma omp single
    {
        #pragma omp task
        { /* esta tarea la ejecuta cualquier hilo */ }

        #pragma omp task
        { /* esta otra tarea también */ }
    }
}
```

<Alert type="info">
El bloque `#pragma omp single` es fundamental: garantiza que solo un hilo genera las tareas. Si todos los hilos generaran tareas simultáneamente, se crearían N copias de cada tarea (una por hilo).
</Alert>

### `#pragma omp taskwait`

Suspende el hilo actual hasta que todas las tareas **hijas directas** del contexto actual hayan completado. Es el equivalente a `pthread_join` para tareas.

```c
#pragma omp task
{ tarea_A(); }

#pragma omp task
{ tarea_B(); }

#pragma omp taskwait
/* aquí tarea_A y tarea_B ya terminaron */
usar_resultados();
```

### `#pragma omp taskgroup`

Similar a `taskwait`, pero espera a todas las tareas creadas dentro del grupo, incluyendo las tareas anidadas (hijas de hijas). Es más potente que `taskwait` para algoritmos recursivos profundos.

```c
#pragma omp taskgroup
{
    #pragma omp task
    procesar_subproblema_recursivo(datos);
}
```

### `#pragma omp task depend`

Declara dependencias de datos entre tareas. El runtime garantiza el orden de ejecución correcto sin que el programador use barreras explícitas.

```c
int x, y, z;

#pragma omp task depend(out: x)
{ x = calcular_x(); }

#pragma omp task depend(in: x) depend(out: y)
{ y = calcular_y(x); }

#pragma omp task depend(in: x, y) depend(out: z)
{ z = calcular_z(x, y); }
```

## Ejercicio 1: Fibonacci recursivo

El número de Fibonacci es el ejemplo canónico para introducir tareas: la recursión genera un árbol de subproblemas que naturalmente se pueden explorar en paralelo.

### Versión secuencial

```c
#include <stdio.h>
#include <stdlib.h>

long fib(int n) {
    if (n < 2) return n;
    return fib(n - 1) + fib(n - 2);
}

int main(void) {
    int n = 40;
    printf("fib(%d) = %ld\n", n, fib(n));
    return 0;
}
```

### Versión paralela con tareas

```c
#include <stdio.h>
#include <stdlib.h>
#include <omp.h>

#define UMBRAL_SECUENCIAL 20

long fib_seq(int n) {
    if (n < 2) return n;
    return fib_seq(n - 1) + fib_seq(n - 2);
}

long fib(int n) {
    if (n < UMBRAL_SECUENCIAL) {
        return fib_seq(n);
    }

    long x, y;

    #pragma omp task shared(x)
    x = fib(n - 1);

    #pragma omp task shared(y)
    y = fib(n - 2);

    #pragma omp taskwait
    return x + y;
}

int main(void) {
    int n = 40;
    long resultado;

    double t_start = omp_get_wtime();

    #pragma omp parallel
    {
        #pragma omp single
        resultado = fib(n);
    }

    double t_end = omp_get_wtime();

    printf("fib(%d) = %ld\n", n, resultado);
    printf("Tiempo: %.4f segundos\n", t_end - t_start);

    return 0;
}
```

```bash
gcc -O2 -fopenmp fib_omp.c -o fib_omp
export OMP_NUM_THREADS=4
./fib_omp
```

<Alert type="warning">
El parámetro `UMBRAL_SECUENCIAL` es crítico. Sin él, `fib(40)` genera más de 300 millones de tareas y el overhead del runtime supera con creces el cómputo. El umbral correcto depende del hardware; experimenta con valores entre 15 y 25.
</Alert>

**Experimento:** mide el tiempo con `UMBRAL_SECUENCIAL` igual a 10, 15, 20 y 25. ¿Qué valor produce el mejor rendimiento en tu máquina?

## Ejercicio 2: Merge Sort paralelo

El merge sort recursivo es un candidato ideal para tareas: cada mitad del arreglo se puede ordenar de forma completamente independiente.

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <omp.h>

#define UMBRAL_INSERCION 1000

void insertion_sort(int *arr, int n) {
    for (int i = 1; i < n; i++) {
        int clave = arr[i];
        int j = i - 1;
        while (j >= 0 && arr[j] > clave) {
            arr[j + 1] = arr[j];
            j--;
        }
        arr[j + 1] = clave;
    }
}

void merge(int *arr, int *tmp, int izq, int mid, int der) {
    int i = izq, j = mid, k = izq;
    while (i < mid && j < der) {
        tmp[k++] = (arr[i] <= arr[j]) ? arr[i++] : arr[j++];
    }
    while (i < mid) tmp[k++] = arr[i++];
    while (j < der) tmp[k++] = arr[j++];
    memcpy(arr + izq, tmp + izq, (der - izq) * sizeof(int));
}

void mergesort(int *arr, int *tmp, int izq, int der) {
    if (der - izq <= UMBRAL_INSERCION) {
        insertion_sort(arr + izq, der - izq);
        return;
    }

    int mid = izq + (der - izq) / 2;

    #pragma omp task shared(arr, tmp) if(der - izq > 10000)
    mergesort(arr, tmp, izq, mid);

    #pragma omp task shared(arr, tmp) if(der - izq > 10000)
    mergesort(arr, tmp, mid, der);

    #pragma omp taskwait
    merge(arr, tmp, izq, mid, der);
}

int main(void) {
    int n = 10000000;
    int *arr = malloc(n * sizeof(int));
    int *tmp = malloc(n * sizeof(int));

    srand(42);
    for (int i = 0; i < n; i++) {
        arr[i] = rand();
    }

    double t_start = omp_get_wtime();

    #pragma omp parallel
    {
        #pragma omp single
        mergesort(arr, tmp, 0, n);
    }

    double t_end = omp_get_wtime();

    int ordenado = 1;
    for (int i = 1; i < n; i++) {
        if (arr[i] < arr[i - 1]) { ordenado = 0; break; }
    }

    printf("Arreglo ordenado: %s\n", ordenado ? "sí" : "no");
    printf("Tiempo: %.4f segundos\n", t_end - t_start);

    free(arr); free(tmp);
    return 0;
}
```

```bash
gcc -O2 -fopenmp mergesort_omp.c -o mergesort_omp
export OMP_NUM_THREADS=4
./mergesort_omp
```

La cláusula `if(der - izq > 10000)` es el **control de granularidad**: cuando el subarreglo es pequeño, la tarea se ejecuta sincrónicamente en el hilo actual en lugar de encolarse. Esto evita el overhead de creación de tareas para trabajo trivial.

## Planificación dinámica

Cuando las iteraciones de un bucle tienen carga desigual, la planificación estática por defecto (`schedule(static)`) produce desbalance: algunos hilos terminan mucho antes que otros. OpenMP ofrece alternativas:

| Schedule | Comportamiento | Cuándo usar |
|---|---|---|
| `static` | Divide iteraciones en bloques iguales asignados antes de ejecutar | Carga uniforme, mejor localidad de caché |
| `dynamic[,chunk]` | Cada hilo pide el siguiente bloque cuando termina el actual | Carga muy variable, iteraciones costosas |
| `guided[,chunk]` | Bloques decrecientes: grandes al inicio, pequeños al final | Carga moderadamente variable |
| `auto` | El compilador/runtime decide | Cuando no sabes qué usar |
| `runtime` | Lee `OMP_SCHEDULE` en tiempo de ejecución | Experimentación sin recompilar |

### Ejercicio 3: Multiplicación de matrices con diferentes schedules

```c
#include <stdio.h>
#include <stdlib.h>
#include <omp.h>

#define N 1024

double A[N][N], B[N][N], C[N][N];

void multiplicar(omp_sched_t tipo, int chunk) {
    omp_set_schedule(tipo, chunk);

    for (int i = 0; i < N; i++)
        for (int j = 0; j < N; j++)
            C[i][j] = 0.0;

    double t_start = omp_get_wtime();

    #pragma omp parallel for schedule(runtime)
    for (int i = 0; i < N; i++) {
        for (int k = 0; k < N; k++) {
            for (int j = 0; j < N; j++) {
                C[i][j] += A[i][k] * B[k][j];
            }
        }
    }

    double t_end = omp_get_wtime();
    printf("schedule tipo=%d chunk=%d -> %.4f s\n", tipo, chunk, t_end - t_start);
}

int main(void) {
    for (int i = 0; i < N; i++)
        for (int j = 0; j < N; j++) {
            A[i][j] = (double)(i + j) / N;
            B[i][j] = (double)(i - j + N) / N;
        }

    multiplicar(omp_sched_static,  0);
    multiplicar(omp_sched_dynamic, 1);
    multiplicar(omp_sched_dynamic, 4);
    multiplicar(omp_sched_guided,  1);

    return 0;
}
```

```bash
gcc -O2 -fopenmp matmul_sched.c -o matmul_sched
export OMP_NUM_THREADS=4
./matmul_sched
```

**Pregunta:** ¿Por qué `schedule(static)` generalmente supera a `schedule(dynamic)` en la multiplicación de matrices? ¿Qué sucede con la caché L1/L2 cuando cambiamos de schedules?

## Secciones paralelas

Las secciones (`sections`) permiten ejecutar bloques de código independientes en paralelo, útiles cuando tienes tareas diferenciadas que no encajan en un bucle.

```c
#include <stdio.h>
#include <stdlib.h>
#include <omp.h>

#define N 1000000

double calcular_suma(double *v, int n) {
    double s = 0.0;
    for (int i = 0; i < n; i++) s += v[i];
    return s;
}

double calcular_max(double *v, int n) {
    double m = v[0];
    for (int i = 1; i < n; i++) if (v[i] > m) m = v[i];
    return m;
}

double calcular_min(double *v, int n) {
    double m = v[0];
    for (int i = 1; i < n; i++) if (v[i] < m) m = v[i];
    return m;
}

int main(void) {
    double *v = malloc(N * sizeof(double));
    for (int i = 0; i < N; i++) v[i] = (double)(i % 1000) / 10.0;

    double suma, maximo, minimo;

    double t_start = omp_get_wtime();

    #pragma omp parallel sections
    {
        #pragma omp section
        suma = calcular_suma(v, N);

        #pragma omp section
        maximo = calcular_max(v, N);

        #pragma omp section
        minimo = calcular_min(v, N);
    }

    double t_end = omp_get_wtime();

    printf("Suma:   %.2f\n", suma);
    printf("Máximo: %.2f\n", maximo);
    printf("Mínimo: %.2f\n", minimo);
    printf("Tiempo: %.4f segundos\n", t_end - t_start);

    free(v);
    return 0;
}
```

<Alert type="info">
Las secciones son eficaces cuando tienes exactamente tantas tareas diferenciadas como hilos disponibles. Si tienes más hilos que secciones, los hilos sobrantes quedan ociosos durante esa región. Para trabajo más dinámico, prefiere tareas.
</Alert>

## Variables de entorno

OpenMP expone su comportamiento a través de variables de entorno, lo que permite ajustar la ejecución sin recompilar.

| Variable | Descripción | Ejemplo |
|---|---|---|
| `OMP_NUM_THREADS` | Número de hilos por defecto | `export OMP_NUM_THREADS=8` |
| `OMP_SCHEDULE` | Schedule y chunk para `runtime` | `export OMP_SCHEDULE="dynamic,4"` |
| `OMP_DYNAMIC` | Permite al runtime ajustar el número de hilos | `export OMP_DYNAMIC=true` |
| `OMP_NESTED` | Habilita paralelismo anidado | `export OMP_NESTED=true` |
| `OMP_MAX_ACTIVE_LEVELS` | Profundidad máxima de paralelismo anidado | `export OMP_MAX_ACTIVE_LEVELS=2` |
| `OMP_PROC_BIND` | Afinidad de hilos a núcleos | `export OMP_PROC_BIND=close` |
| `OMP_PLACES` | Define los lugares de ejecución | `export OMP_PLACES=cores` |

```bash
export OMP_NUM_THREADS=4
export OMP_SCHEDULE="guided,2"
export OMP_DYNAMIC=false

./matmul_sched
```

<Alert type="tip">
Para producción, fija `OMP_DYNAMIC=false` y define explícitamente `OMP_NUM_THREADS`. El ajuste dinámico del runtime puede provocar variabilidad en los tiempos que dificulta el benchmarking reproducible.
</Alert>

## Ejercicios propuestos

### EP1: Árbol de suma con tareas y dependencias

Dado un arreglo de N números, implementa una reducción en árbol binario usando `#pragma omp task depend`. Cada tarea suma dos elementos y almacena el resultado en un arreglo auxiliar. Las dependencias deben garantizar que cada nivel del árbol espere al nivel anterior sin usar `taskwait` explícito. Compara el tiempo contra `reduction(+:suma)` estándar.

### EP2: QuickSort paralelo adaptativo

Implementa QuickSort paralelo usando tareas OpenMP. Aplica una heurística adaptativa: usa tareas solo cuando el subarreglo supere un umbral configurable, y caer en `std::sort` (o inserción) para subarreglos pequeños. Mide el speedup para arreglos de 1M, 10M y 100M elementos con 2, 4 y 8 hilos.

### EP3: Pipeline de procesamiento de imágenes

Modela un pipeline de tres etapas para procesar una secuencia de imágenes (matrices N×N): (1) aplicar un filtro de suavizado, (2) detectar bordes con el operador de Sobel, (3) binarizar el resultado. Usa `#pragma omp sections` para ejecutar las etapas en paralelo con doble buffer (mientras la etapa 1 procesa la imagen i+1, la etapa 2 procesa la imagen i). Reporta el throughput en imágenes por segundo.
